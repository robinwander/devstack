# Devstack Dashboard — UX Redesign Vision

**Date**: 2025-03-08
**Prerequisite**: [UX Audit](./UX_AUDIT.md)
**Scope**: Complete visual and interaction redesign of the devstack dashboard

---

## 1. Design Philosophy

### What Is This App, Really?

Devstack is a **log viewer that also shows service health**. That's it. Everything else is secondary.

A developer opens this dashboard for one of three reasons:
1. **"Is everything OK?"** — a 2-second health glance
2. **"Something broke, show me the error"** — find, read, and copy an error
3. **"Let me watch what's happening"** — stream logs while developing

Every design decision must serve these three workflows. If a pixel doesn't help one of them, it goes.

### Design Principles

**1. Logs are the product.** The log viewer gets maximum screen real estate, maximum readability, and maximum care. Everything else is a supporting actor. The sidebar, header, toolbar — these exist to help you filter and navigate logs. They should be as invisible as possible until needed.

**2. Readability over aesthetics.** A developer stares at log lines for hours. Contrast ratios, line height, font sizing, and color coding are not cosmetic decisions — they're ergonomic ones. We optimize for sustained reading, not first-impression screenshots.

**3. Hierarchy through information design, not opacity.** The current UI has 12+ opacity levels trying to communicate hierarchy. This is a failure of design. Hierarchy comes from: what's bigger, what's bolder, what's colored, and what has more space around it. We use 3-4 text colors and let typography, size, and spatial position do the rest.

**4. Color is semantic, not decorative.** Every color means something. Green = ready. Red = error. Amber = warning. Service colors identify services. There is no decorative color — no glows, no gradients, no tinted backgrounds for visual interest. Color earns its place by carrying information.

**5. Progressive disclosure.** Start sparse, get dense on demand. The default view shows health + logs. Filters, facets, service details, and actions reveal themselves when needed — through search, keyboard shortcuts, hover states, or explicit toggles. No permanent sidebars eating space for rarely-used information.

## 2. Aesthetic Direction

### Killing the AI Slop

The current dashboard is a textbook "AI dark terminal" — teal-on-black, glow effects, noise textures, zero border radius, monospace everything. It looks like it was generated by prompting "make a developer dashboard." We burn all of it.

### The New Direction: "Instrument-Grade Clarity"

Think of the best physical instruments — a Leica camera's viewfinder, a Fluke multimeter's display, an oscilloscope's trace screen. These are tools built for sustained professional use. They share traits:

- **High contrast, low decoration.** The information is the interface.
- **Confidence in negative space.** They don't fill every pixel. They let the reading breathe.
- **Semantic density.** Lots of information, but organized so your eye knows exactly where to go.
- **Material honesty.** They don't pretend to be something they aren't. No faux-3D, no unnecessary ornamentation.

Our dashboard should feel like a **precision instrument designed by someone who actually reads logs all day.** Not a "terminal aesthetic" cosplay, but a genuine tool optimized for the task.

### What Makes This Distinctive

The memorable quality of this dashboard will be: **how readable the logs are.** When someone uses devstack after using other log viewers, they should feel physical relief. The typography is tuned for scanning. The color coding makes service identification instant. The search is fast and the results are clear. The information density is high but never overwhelming.

This isn't flashy. It's the design equivalent of a perfectly tuned mechanical keyboard — you notice the quality through sustained use, not from a screenshot.

### Tone Keywords

`precise` · `confident` · `warm-neutral` · `dense-but-clear` · `tool-not-toy`

## 3. Color System

### Philosophy

The UI chrome is neutral. The content provides the color. Service colors are the stars. Status colors carry meaning. The accent color is quiet and purposeful.

### Palette Architecture

```
┌─────────────────────────────────────────────┐
│  SURFACES (90% of pixels)                   │
│  Warm neutral backgrounds and borders       │
│  Light mode: warm white → warm gray          │
│  Dark mode: warm charcoal → warm dark        │
├─────────────────────────────────────────────┤
│  TEXT (hierarchy through weight + size)      │
│  3 levels: primary, secondary, tertiary     │
│  No opacity tricks. Defined colors only.    │
├─────────────────────────────────────────────┤
│  SEMANTIC (status + log levels)             │
│  green (ready), amber (warning/starting),   │
│  red (error/failed), blue (info)            │
├─────────────────────────────────────────────┤
│  SERVICE COLORS (log identification)        │
│  8 perceptually-balanced hues in OKLCH      │
│  Used boldly: backgrounds, tab accents,     │
│  sidebar indicators, log row tints          │
├─────────────────────────────────────────────┤
│  INTERACTIVE (one quiet accent)             │
│  Focus rings, active states, links          │
│  Not competing with service/status colors   │
└─────────────────────────────────────────────┘
```

### Light Mode First, Dark Mode Excellent

Light mode is the default. Dark mode is not an inverted afterthought — it's a carefully designed alternative. Both modes are first-class citizens built from the same token system.

Why light mode first? Because:
- Most developers work in mixed environments (light IDE, light browser, light docs)
- Light backgrounds provide better contrast ratios with less effort
- It forces us to design with real hierarchy instead of hiding behind "dark is cool"
- Dark mode is trivially broken when hard-coded dark colors leak in; light-first prevents this

Dark mode follows the "lighter surfaces = higher elevation" convention. No pure black. No glow effects. Depth comes from subtle surface color shifts, not shadows or glows.

### Neutral Palette (Warm Tint)

All neutrals carry a subtle warm tint (OKLCH hue ~260 for cool-blue in dark mode, ~50 for warm-sand in light mode). Zero chroma grays are banned — they feel dead.

```css
/* Light mode surfaces */
--surface-base:     oklch(0.985 0.005 55);   /* warm off-white */
--surface-raised:   oklch(1.000 0.000 0);     /* pure white for cards */
--surface-sunken:   oklch(0.965 0.007 55);   /* warm light gray */
--surface-overlay:  oklch(1.000 0.000 0);     /* white overlays */

/* Dark mode surfaces */
--surface-base:     oklch(0.130 0.006 260);   /* warm charcoal */
--surface-raised:   oklch(0.160 0.007 260);   /* slightly lighter */
--surface-sunken:   oklch(0.100 0.005 260);   /* slightly darker */
--surface-overlay:  oklch(0.180 0.008 260);   /* popover/modal */
```

### Text Hierarchy (3 Levels, No Opacity)

```css
/* Light mode */
--text-primary:    oklch(0.15 0.005 260);    /* near-black, warm */
--text-secondary:  oklch(0.40 0.010 260);    /* confident gray */
--text-tertiary:   oklch(0.58 0.008 260);    /* readable de-emphasis */

/* Dark mode */
--text-primary:    oklch(0.93 0.005 80);     /* warm off-white */
--text-secondary:  oklch(0.65 0.008 260);    /* clear mid-tone */
--text-tertiary:   oklch(0.48 0.006 260);    /* readable de-emphasis */
```

No text ever uses an opacity modifier. These three colors handle all hierarchy needs. If you need more levels, you're designing the wrong hierarchy — use font size or weight instead.

### Semantic Colors

Tuned for both light and dark backgrounds. Each semantic color has a base (for text/icons), a tint (for backgrounds), and a border variant.

```css
/* Success / Ready */
--semantic-green:       oklch(0.55 0.16 155);
--semantic-green-tint:  oklch(0.55 0.16 155 / 8%);
--semantic-green-bold:  oklch(0.65 0.19 155);  /* dark mode text */

/* Warning / Starting */
--semantic-amber:       oklch(0.65 0.16 75);
--semantic-amber-tint:  oklch(0.65 0.16 75 / 8%);
--semantic-amber-bold:  oklch(0.78 0.14 75);

/* Error / Failed */
--semantic-red:         oklch(0.55 0.20 25);
--semantic-red-tint:    oklch(0.55 0.20 25 / 8%);
--semantic-red-bold:    oklch(0.70 0.18 25);

/* Info (used sparingly) */
--semantic-blue:        oklch(0.55 0.14 255);
--semantic-blue-tint:   oklch(0.55 0.14 255 / 8%);
```

### Service Colors (8 Perceptually Balanced Hues)

These are the visual identity of the log viewer. They must be:
- **Perceptually equal in weight** — no color should "pop" more than another
- **Distinct from semantic colors** — a service's blue must not be confused with the info blue
- **Readable as text** — service name labels use these colors directly
- **Beautiful as backgrounds** — row tints use these at 5-7% opacity

```css
/* All at L=0.62, C=0.14 for perceptual balance */
--svc-0: oklch(0.62 0.14 240);   /* Steel blue */
--svc-1: oklch(0.62 0.14 300);   /* Iris */
--svc-2: oklch(0.62 0.14 60);    /* Bronze */
--svc-3: oklch(0.62 0.14 15);    /* Coral */
--svc-4: oklch(0.62 0.14 175);   /* Sage */
--svc-5: oklch(0.62 0.14 335);   /* Orchid */
--svc-6: oklch(0.62 0.14 125);   /* Moss */
--svc-7: oklch(0.62 0.14 275);   /* Plum */
```

Service color assignment is **deterministic** — derived from a hash of the service name, not array index. "api" is always steel blue. Across runs, across sessions, across machines.

### Interactive Accent

One quiet color for focus rings, active states, and interactive affordances. Not competing with service or status colors.

```css
--accent:           oklch(0.55 0.12 255);   /* quiet blue */
--accent-hover:     oklch(0.50 0.14 255);
--accent-ring:      oklch(0.55 0.12 255 / 40%);
```

## 4. Typography

### Philosophy

**Two fonts, used with intention.** The sans-serif font handles all UI chrome. The monospace font handles all data content. There is no ambiguity about which font to use — if it's a UI element (button, label, heading, tab), it's sans. If it's data from the system (log content, timestamps, URLs, ports, run IDs), it's mono.

### Font Selection

**UI Font: Outfit Variable** — Keep it. Outfit is a modern geometric sans with excellent readability at small sizes. It's not Inter/Roboto (overused) and has good personality. But we'll use it with more intention — dramatic weight contrast, clear size hierarchy.

**Data Font: JetBrains Mono** — Keep it. JetBrains Mono is one of the best programming fonts. It has excellent character distinction (0/O, 1/l/I), proper ligatures, and tabular figures built in.

### Type Scale

A constrained 5-step scale with clear purpose for each level:

```css
--text-xs:    0.6875rem;  /* 11px — metadata, timestamps, line numbers */
--text-sm:    0.8125rem;  /* 13px — log content, secondary UI */
--text-base:  0.875rem;   /* 14px — body text, primary UI */
--text-lg:    1.0rem;     /* 16px — section headings */
--text-xl:    1.25rem;    /* 20px — page headings */
```

Why this scale and not a standard 12/14/16/18/24? Because this is a **dense information tool**. Standard web scales waste vertical space. A log viewer at 16px base is wasteful — 14px is the right base for sustained reading of structured data. 13px for the actual log content maximizes visible lines while maintaining readability.

### Weight Usage

```
Outfit 700 (Bold)       — Page headings, empty state titles
Outfit 600 (Semibold)   — Section headings, active tab labels, service names
Outfit 500 (Medium)     — Buttons, labels, primary UI text
Outfit 400 (Regular)    — Body text, secondary descriptions

JetBrains Mono 500      — Log content, search input
JetBrains Mono 400      — Timestamps, metadata, run IDs
```

### Monospace Boundaries (Strict Rule)

Monospace is used ONLY for:
- Log line content
- Timestamps
- Run IDs
- Port numbers and URLs
- Code snippets in empty states

Everything else is Outfit. The search input, facet labels, tab names, button text, section headers, stack names — all sans-serif. The current 80% monospace usage drops to ~40%.

## 5. Layout Architecture

### The Core Insight

The current layout is: `[Sidebar 260px] + [Facets 256px] + [Logs]`

On a 1440px screen, logs get only 924px — **64% of the viewport for the thing that IS the product.** With facets open on a smaller screen, it's even worse.

The new layout is: `[Sidebar ~160px, collapsible] + [Logs 100%]`

Facets become a popover. The sidebar shrinks dramatically. Logs get maximum width.

### Desktop Layout (≥1024px)

```
┌──────────────────────────────────────────────────────────────────────┐
│  HEADER                                                              │
│  [Logo] [Stack ▾ project-name] [● ● ● 3/3 ready]        [Daemon ●] │
├─────────┬────────────────────────────────────────────────────────────┤
│ SIDEBAR │  TOOLBAR                                                   │
│         │  [All] [api] [web] [worker]    [🔍 Search...] [⏱ ▾] [⊞]  │
│ ● api   ├────────────────────────────────────────────────────────────┤
│   ready │  LOG VIEWER                                                │
│         │                                                            │
│ ● web   │  42  19:24:56.104  api     Task completed: cache-warmup    │
│   ready │  43  19:24:57.265  api     GC pause: 4ms                   │
│         │  44  19:24:58.818  web     GET / - 200 in 29ms             │
│ ● worker│  45  19:24:58.941  api     GET /health HTTP/1.1            │
│   ready │  46  19:24:59.315  web     GET /users - 200 in 24ms        │
│─────────│  47  19:24:59.523  api     Health check passed             │
│ GLOBALS │  48  19:25:00.125  web     GET / HTTP/1.1                  │
│ ● mysql │  49  19:25:01.673  api     Memory usage at 78% threshold   │
│ ● cache │  50  19:25:02.559  web     GET /dashboard - 200 in 4ms     │
│ ● db    │  ...                                                       │
│         │                                                            │
│ ────────│                                            [↓ 12 new]      │
│ /tmp/.. │                                                            │
└─────────┴────────────────────────────────────────────────────────────┘
```

**Sidebar**: ~160px. Shows service name + status. No URLs, no action buttons in the default view. Hovering a service reveals a popover with URL, port, and actions (copy, open, restart). Clicking a service filters logs to that service. The sidebar can be collapsed to just status dots (~40px) via a toggle or keyboard shortcut.

**Toolbar**: Single row combining service tabs + search + time range + filter toggle. Compact and cohesive. The tab bar and search bar share one horizontal strip.

**Log Viewer**: Takes all remaining space. This is the product.

### Sidebar Detail: Hover Popover

When hovering over a service in the sidebar, a popover appears to the right showing:

```
┌──────────────────────────┐
│  api                     │
│  ● Ready                 │
│                          │
│  http://localhost:38831   │
│  [Copy URL] [Open ↗]    │
│                          │
│  [Restart ↻]             │
└──────────────────────────┘
```

This replaces the current always-visible-but-hard-to-discover action buttons. The popover appears on hover with a 200ms delay (to avoid accidental triggers) and stays open while the mouse is in the popover area.

### Sidebar Detail: Stack Actions

Stop and Kill are NOT in the sidebar. They move to:
1. The **header** — a dropdown menu next to the stack name with "Stop stack" and "Force kill" options
2. The **command palette** — accessible via ⌘K
3. The right side of the sidebar header as a subtle "⋯" overflow menu

Kill always requires confirmation. Stop does not (it's graceful and non-destructive).

### Tablet Layout (768px–1023px)

Sidebar collapses to icon-only mode (~40px) showing just status dots. Tapping a dot opens the service popover. Everything else same as desktop.

### Mobile Layout (<768px)

No sidebar. Services are accessible only through the tab bar and command palette. The slide-over drawer pattern is available via a header button for full service details, but the default view is logs-only.

### Facets: Popover, Not Column

The facets panel is a **popover** triggered by a filter icon in the toolbar. It opens as a ~320px wide dropdown anchored to the filter button, overlaying the log viewer. It does NOT consume layout space.

The popover contains the same facet sections (service, level, stream, custom fields) but in a more compact vertical layout. Active filters show as chips in the toolbar so the user always sees what's filtered even when the popover is closed.

```
TOOLBAR: [All] [api] [web]  [🔍 Search...] [service:api ✕] [level:error ✕] [⊞ Filters] [⏱ Live ▾]
```

When filters are active, they appear as removable chips between the search bar and the filter button. This makes active filters visible without consuming permanent column space.

## 6. Log Viewer Redesign

This is the most important section. The log viewer is the product.

### Information Hierarchy per Line

Every log line communicates a stack of information. Here's the priority order:

```
1. ERROR/WARN status   (most important — is this an error?)
2. Log message content  (what happened?)
3. Service identity     (which service?)
4. Timestamp            (when?)
5. Line number          (where in the sequence?)
```

The current design treats all of these at nearly equal visual weight (small monospace text, low opacity). The new design creates clear hierarchy:

### Log Line Anatomy

```
│▌│ 42 │ 19:24:56.104 │ api │ Failed to connect to upstream service: connection refused │
 ↑   ↑        ↑          ↑     ↑
 │   │        │          │     └─ Content: text-sm mono, text-primary for errors,
 │   │        │          │        text-secondary for info
 │   │        │          │
 │   │        │          └─ Service: text-sm sans, semibold, service color
 │   │        │
 │   │        └─ Timestamp: text-xs mono, text-tertiary, tabular-nums
 │   │
 │   └─ Line number: text-xs mono, text-tertiary, right-aligned
 │
 └─ Service color strip: 3px left border, service color
```

### Error/Warn Treatment

Error and warning lines get distinct treatment that makes them **impossible to miss** during casual scanning:

**Error lines:**
- Left strip turns red (overrides service color)
- Full row gets a subtle red tint background (`--semantic-red-tint`)
- Content text uses `--semantic-red-bold` (not just "reddish gray")
- The service name is still visible in its service color (no information loss)

**Warning lines:**
- Left strip turns amber
- Subtle amber tint background
- Content text uses `--semantic-amber-bold`

**Info lines (default):**
- Service color strip
- No background tint (or very subtle service-color tint in "All" view for visual grouping)
- Content text in `--text-secondary` — visible but not competing with errors

### Service Color in "All" View

When viewing all services interleaved, service identity must be instantly scannable. The current design uses a 3px color strip + colored service name. The new design amplifies this:

- **3px left border** in service color (keep)
- **Service name** in service color, semibold, sans-serif (amplify — currently mono, hard to scan)
- **Subtle row background tint** in service color at 4% opacity (keep, but make more visible)
- **Service name shown on every line** — the current "hide repeated service names" trick saves a few pixels but makes scanning harder. When I'm looking for "api" lines in a stream of mixed logs, I need the label on every line. **Show it always.**

Wait — there's a counter-argument: showing the service name on every single line creates visual noise when 20 consecutive lines are all from the same service. The "hide repeats" approach reduces that noise. 

The solution: **Show the service name on every line, but at reduced weight for consecutive same-service lines.** The first line of a service "run" shows the name in full semibold color. Subsequent same-service lines show it in `text-tertiary` weight — still readable if you look, but visually quieter.

### Single-Service View

When filtering to a single service, the layout changes:
- No service color strip (unnecessary — you know which service you're looking at)
- No service name column (same reason — saves ~100px of horizontal space)
- Service identity is communicated by the active tab color
- Log content gets more horizontal space

### Contrast Ratios (Non-Negotiable)

Every text element must meet WCAG AA (4.5:1 for normal text, 3:1 for large text):

| Element | Current Ratio | Target Ratio |
|---------|--------------|--------------|
| Line numbers | ~1.5:1 ❌ | ≥4.5:1 |
| Timestamps | ~2.0:1 ❌ | ≥4.5:1 |
| Info log content | ~3.2:1 ❌ | ≥4.5:1 |
| Error log content | ~4.8:1 ✓ | ≥4.5:1 |
| Service names | varies | ≥4.5:1 |
| Search placeholder | ~1.8:1 ❌ | ≥4.5:1 |

### Line Height and Density

Log viewers live and die by information density. The goal: **maximum visible lines without sacrificing readability.**

```css
.log-line {
  line-height: 1.5;        /* 19.5px at 13px font — readable without crowding */
  padding: 2px 0;          /* 4px total vertical padding */
  /* Total line height: ~24px — fits ~33 lines in an 800px viewport */
}
```

Compare to the current design which uses `py-[3px]` (6px padding) with similar font size, giving ~28px per line and ~28 visible lines. The new design shows **~18% more log lines** in the same viewport.

### Expanded Log Detail

Clicking a log line expands a detail panel below it. The new design:

- **No JSON editor library.** Use a simple syntax-highlighted `<pre>` with indented JSON. `vanilla-jsoneditor` is overkill for read-only viewing and creates visual inconsistency.
- **Clean metadata header**: timestamp, service, stream, level — all on one line, clear typography
- **Copy button**: "Copy line" for plain text, "Copy JSON" for structured logs — clearly labeled
- **Share button**: If agent session is active, share button appears. Otherwise hidden.
- **Smooth expansion**: Uses `grid-template-rows: 0fr → 1fr` transition, not height animation. Plays well with the virtualizer.

### Search Redesign

The search bar is the second most important UI element after the log content itself.

**Current problems:**
- Monospace font in the search input (why?)
- Regex toggle is a separate button visually disconnected from the input
- Search match navigation (up/down arrows) is cramped
- Suggestion dropdown has edge cases and race conditions

**New design:**

The search input uses the sans-serif font. It occupies the right portion of the toolbar row, expanding on focus.

```
┌──────────────────────────────────────────────────────────────┐
│ [All] [api] [web] [worker]     [🔍 Search logs...] [⏱ ▾] [⊞]│
└──────────────────────────────────────────────────────────────┘

  When focused and searching:

┌──────────────────────────────────────────────────────────────┐
│ [All] [api] [web] [worker]  [🔍 connection refused  1/3 ▲▼✕]│
└──────────────────────────────────────────────────────────────┘
```

- Match count and navigation arrows appear **inside** the input (like Chrome's find bar)
- Regex mode toggle is inside the input as a small icon, not a separate button
- Clearing search is a ✕ inside the input
- Keyboard shortcut hint (`/`) shown in the search input when empty, disappears on focus

**Search suggestions** use the native popover API for positioning. No `setTimeout` hacks for blur handling. The dropdown is clean, fast, and well-positioned.

## 7. Header Redesign

### Current Problems
- Logo is a nested-rectangle SVG that means nothing
- "DEVSTACK" in tiny uppercase tracking-widest text — trying too hard to be minimal
- Health bars are cryptic 1.5px rectangles
- Run selector doesn't show enough information

### New Header

```
┌──────────────────────────────────────────────────────────────────┐
│  ⬡ devstack    dev ▾ devstack-demo    ● 3/3 ready        ● API │
└──────────────────────────────────────────────────────────────────┘
```

**Logo**: Simple wordmark. "devstack" in Outfit Semibold, one size up from body. No icon. No nested rectangles. The product name IS the logo. It's a developer tool — developers respect simplicity.

**Stack Selector**: Stack name in semibold + project name in secondary text + chevron. Clicking opens the run dropdown. The dropdown now shows:
- Per-run: stack name, project name, creation time, service count, health summary (e.g., "3/3 ready" or "1 failed")
- Clear separation between active and stopped runs
- "Clean up old runs" at the bottom

**Health Summary**: Replace the cryptic bars with a simple text indicator: `● 3/3 ready` in green, or `● 1/3 failed` in red, or `● 2/3 starting` in amber. The colored dot communicates status at a glance; the text provides precision. The bars added nothing that the text doesn't communicate better.

**Daemon Status**: `● API` or `● Connected` in the right corner. When disconnected: `⚠ Disconnected` with a retry action.

**Stack Actions**: A subtle "⋯" button next to the stack selector that opens a dropdown with "Stop stack", "Force kill" (with confirmation), "Copy run ID", "Copy dashboard URL".

### Header Height

Current: 48px (`h-12`). New: 44px. Shave 4px — every pixel of vertical space given to the log viewer matters.

## 8. Sidebar Redesign

### Current Problems
- 260px wide — too much space for too little information
- Run ID at top serves no purpose for most users
- Action buttons are hard to discover (60% opacity)
- URLs shown at 11px in nearly invisible color
- Globals section is visually orphaned

### New Sidebar: Compact Service Navigator

**Width: 160px** (collapsed: 40px, just dots)

The sidebar is a simple vertical list of services with their status. That's all it needs to be in its default state.

```
┌──────────┐
│ SERVICES │
│          │
│  ● api   │
│  ● web   │
│  ● worker│
│          │
│ GLOBALS  │
│  ● mysql │
│  ● cache │
│  ● db    │
│          │
│──────────│
│ /tmp/... │
└──────────┘
```

Each row: status dot + service name. That's it. Clean, scannable, dense.

**On hover**, a popover card appears to the right of the row:

```
                  ┌─────────────────────────┐
│  ● api   │ ←→  │  api — Ready             │
│          │      │  http://localhost:38831   │
                  │  Port: 38831             │
                  │                          │
                  │  [Copy URL] [Open] [↻]   │
                  └─────────────────────────┘
```

This eliminates the need for always-visible action buttons and URLs. The information appears exactly when you need it, in a clean, readable format.

**Clicking a service** filters logs. The active service gets a colored left border (service color) and a subtle background tint. Same as now, but cleaner.

**Collapse toggle**: A small button at the top of the sidebar or a keyboard shortcut (`[` or `]`) collapses it to 40px — just status dots aligned vertically. This gives even more space to logs.

### Globals Treatment

Globals are visually separated by a label + divider, same as now but cleaner. They show in the hover popover just like regular services (with URL/port if available). They don't appear in the log tab bar (they don't generate logs).

### Stack Actions (Moved Out)

Stop and Kill are no longer in the sidebar. They live in the header's stack overflow menu and the command palette. This removes the confusing tiny icon-only buttons for destructive actions.

## 9. Toolbar Redesign

### Current Problems
- Two rows of controls competing for attention (search bar row + tab bar row)
- Time range buttons are cramped and ambiguous
- Facets toggle is just another icon in a sea of icons
- Line count "500" shown with no context

### New Toolbar: Single Unified Row

Everything lives in one horizontal strip:

```
┌──────────────────────────────────────────────────────────────────┐
│ [● All] [● api] [● web] [● worker]  ║ [🔍 Search...] [⏱▾] [⊞] │
└──────────────────────────────────────────────────────────────────┘
```

**Left side**: Service tabs. Each tab shows a status dot + name. The active tab gets an underline in the service's color (or the accent color for "All"). Tabs scroll horizontally with fade indicators if they overflow.

**Right side**: Search input + Time range dropdown + Filter button.

**Time range**: A single dropdown button showing the current selection ("Live", "Last 5m", "Last 15m", "Last 1h"). Not 4 separate radio buttons. "Live" is the default and shows a subtle pulse indicator.

**Filter button**: Opens the facets popover. When filters are active, the button shows a badge count and active filters appear as chips:

```
┌──────────────────────────────────────────────────────────────────┐
│ [● All] [● api]  [level:error ✕]  ║ [🔍 Search...] [⏱ Live▾] [⊞2]│
└──────────────────────────────────────────────────────────────────┘
```

**Line count**: Shown in the toolbar right-edge, styled as metadata: "247 lines" in `text-tertiary`. Provides context for the current view without competing for attention.

### Keyboard Shortcut Hints

The current permanent overlay at the bottom-left is removed. Instead:
- Shortcuts are shown in the command palette (⌘K) which lists all available actions with their shortcuts
- The search input shows `/` as a placeholder hint when empty
- On first visit, a small dismissible tooltip appears near the search bar: "Tip: Press / to search, E for errors, ⌘K for all shortcuts"
- After dismissal, hints are gone forever (stored in localStorage)

## 11. Interaction & Motion

### Philosophy

Motion serves function, not decoration. The three legitimate uses of animation in this app:

1. **State transitions** — something changed, the animation shows what changed
2. **Spatial orientation** — something opened/closed, the animation shows where it went
3. **Attention direction** — something needs attention, a subtle pulse or highlight draws the eye

That's it. No stagger animations on service lists. No noise textures. No glow effects. No `whileHover: { x: 2 }` imperceptible nudges.

### Timing Tokens

```css
--duration-instant:  100ms;   /* button press, toggle */
--duration-fast:     150ms;   /* tooltip, hover state */
--duration-normal:   200ms;   /* popover, dropdown */
--duration-slow:     300ms;   /* sidebar collapse, panel expand */

--ease-out:          cubic-bezier(0.16, 1, 0.3, 1);    /* elements entering */
--ease-in:           cubic-bezier(0.7, 0, 0.84, 0);    /* elements leaving */
--ease-in-out:       cubic-bezier(0.65, 0, 0.35, 1);   /* state toggles */
```

### Specific Animations

**Sidebar collapse/expand**: Width transition over `--duration-slow` with `--ease-in-out`. Content fades as width shrinks.

**Facets popover**: Scale from 0.97 + opacity from 0 on enter (150ms, ease-out). Opacity to 0 on exit (100ms, ease-in).

**Log detail expand**: `grid-template-rows: 0fr → 1fr` over 200ms. No janky height animation.

**New logs indicator**: Slides up from bottom of log viewer, `--duration-normal`, `--ease-out`.

**Run switching**: Cross-fade the log viewer content only (150ms). The sidebar and header remain stable — no full-page opacity flash.

**Status dot pulse**: Only for "starting" state. Gentle opacity pulse (2s cycle). Stops when state changes. Removed for all other states.

**Scroll controls**: "↓ 12 new lines" button uses `--duration-normal` to slide in when new logs arrive while scrolled up. Disappears instantly when clicked (no exit animation — get out of the way).

### Reduced Motion

All spatial animations (slides, scales) are replaced with simple opacity fades. Pulse animations are disabled. Transitions still occur but without movement.

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 150ms !important;  /* Keep transitions, just no motion */
  }
}
```

## 12. Border Radius

### Current: `--radius: 0` (Everything Sharp)

This is the most visible "vibe code" decision. Zero radius on everything creates a hostile, unfinished aesthetic that requires exceptional spatial design to pull off. This codebase doesn't have that.

### New: Subtle, Consistent Radius

```css
--radius-sm:   4px;    /* buttons, inputs, badges, chips */
--radius-md:   6px;    /* cards, popovers, dropdowns */
--radius-lg:   8px;    /* modals, large containers */
--radius-full: 9999px; /* status dots, pills */
```

The radius is **small and consistent**. Not the large, bubbly 12-16px of consumer apps. Not the zero of "hacker terminal." A professional 4-6px that says "this was designed, not defaulted."

## 13. Spacing System

### Current Problems
- `px-3 py-2.5` repeated everywhere — monotonous
- No clear spatial rhythm
- Inconsistent gaps between similar elements

### New: 4px Base Grid

```css
--space-1:   4px;
--space-2:   8px;
--space-3:   12px;
--space-4:   16px;
--space-6:   24px;
--space-8:   32px;
--space-12:  48px;
--space-16:  64px;
```

### Spatial Rules

**Grouping**: Related items have tight spacing (4-8px). Unrelated groups have generous spacing (16-24px). This creates visual grouping without needing borders or containers.

**Sections**: Section headers have more space above than below — the header "belongs to" its content, pulling it downward.

```css
.section-header {
  margin-top: var(--space-6);    /* 24px above */
  margin-bottom: var(--space-2); /* 8px below — attached to content */
}
```

**Component internals**: Buttons use 8px vertical, 12px horizontal padding. Inputs use 8px vertical, 12px horizontal. List items use 6-8px vertical, 12-16px horizontal. Consistency within component types, variety between different types.

**Sidebar vs. log viewer**: The sidebar is relatively spacious (12px horizontal padding, 8px between items). The log viewer is dense (minimal padding, tight line spacing). This contrast reinforces the sidebar's role as navigation (scan) vs. the log viewer's role as content (read).

## 14. Keyboard & Accessibility

### Keep What Works

The existing keyboard system is excellent. Keep it all:
- `/` or `Ctrl+F` → focus search
- `Escape` → clear/unfocus
- `E` → toggle error filter
- `W` → toggle warning filter
- `F` → toggle facets
- `1-9` → switch service tabs
- `⌘K` → command palette
- `⌘G` / `Shift+⌘G` → next/prev search match

### Add

- `[` or `]` → collapse/expand sidebar
- `J` / `K` → scroll up/down by log line (vim-style)
- `Enter` on focused log line → expand detail
- `Escape` from expanded detail → collapse it

### Focus Management

- Tab order: Header → Sidebar → Toolbar → Log viewer
- Focus rings: 2px `--accent` outline with 2px offset. Visible in both light and dark modes. Never removed, always via `:focus-visible`.
- Skip link: Hidden "Skip to logs" link that appears on Tab, jumps focus to the log viewer.
- Log viewer: `role="log"` with `aria-live="polite"` for new entries (existing, keep it).

### Screen Reader Announcements

- When filters change: announce "Showing X log lines from [service] at [level]"
- When search matches: announce "X matches found"
- When health changes: announce "Service [name] is now [state]"

## 15. Implementation Strategy

### Phase 1: Foundation (Design Tokens + Layout)
**Goal**: Establish the new token system and layout architecture. No visual polish yet — just get the structure right.

1. Define all CSS custom properties (colors, typography, spacing, radius, motion)
2. Implement light/dark mode token switching with `prefers-color-scheme` and manual toggle
3. Restructure layout: narrow sidebar + full-width log viewer
4. Remove all hard-coded colors — everything through tokens
5. Remove decorative elements: noise texture, header glow, status glow effects, stagger animations

**Deliverable**: The app looks "boring" but correct. All layout measurements are right, all colors come from tokens, both themes work.

### Phase 2: Log Viewer (The Core Product)
**Goal**: Make the log viewer world-class.

1. Fix all contrast ratios to meet WCAG AA
2. Implement new log line anatomy: typography, spacing, service color treatment
3. Redesign expanded log detail (drop vanilla-jsoneditor, use syntax-highlighted `<pre>`)
4. Implement deterministic service color hashing
5. Always-visible service names with reduced-weight consecutive lines
6. Optimize line density (target: 33+ visible lines in 800px)
7. New search bar design (match count inside input, clean suggestions)

**Deliverable**: The log viewer is excellent. A developer opens it and thinks "this is so much better."

### Phase 3: Chrome (Sidebar, Header, Toolbar)
**Goal**: Polish everything around the log viewer.

1. Redesign sidebar: 160px, hover popovers for service details
2. Redesign header: wordmark, improved run selector, text health summary
3. Redesign toolbar: single row, time range dropdown, filter chips
4. Facets as popover instead of column
5. Move stack actions to header overflow menu
6. Add sidebar collapse/expand

**Deliverable**: The full app is cohesive and polished.

### Phase 4: Empty States + Onboarding
**Goal**: Nail the first impression.

1. Redesign empty dashboard with informative stack cards
2. First-visit keyboard shortcut hint (dismissible)
3. Improved error boundary with copy/report/recover

### Phase 5: Animation + Delight
**Goal**: Add purposeful motion and final polish.

## 16. What We're NOT Doing

Explicit scope boundaries to prevent scope creep:

- **No light/dark mode toggle in v1.** We implement both modes correctly via `prefers-color-scheme`. A manual toggle is a nice-to-have for later.
- **No custom themes.** The token system supports it, but we ship one light and one dark theme.
- **No redesigned command palette.** The existing command palette works. We add missing actions (restart) but don't redesign the UI.
- **No sound design.** This is a silent tool.
- **No illustrations or custom icons.** Lucide icons are fine. No custom illustration work.
- **No responsive below 375px.** We support down to 375px (iPhone SE). Below that, graceful degradation.

## 17. Success Criteria

How we know the redesign succeeded:

1. **The Slop Test**: Show the dashboard to 5 developers and ask "did AI make this?" If more than 1 says yes, we failed.
2. **The Readability Test**: Can a developer scan 50 log lines and find the one error in under 3 seconds?
3. **The Density Test**: Does the redesigned log viewer show at least 15% more visible lines than the current one at the same viewport size?
4. **The Glance Test**: Can a developer determine "is my stack healthy?" in under 2 seconds without reading any text?
5. **The Switching Test**: Can a developer switch to this from their terminal log viewer and not want to switch back?
6. **The Contrast Test**: Every text element passes WCAG AA (4.5:1).
7. **The Light Test**: Does the app look equally good in light and dark mode?
8. **The Width Test**: On a 1440px screen with the sidebar open, does the log viewer get at least 80% of the remaining horizontal space?
